# 项目拷打器

## 解决的核心痛点
用户在面试中被追问过往经历和项目细节时，常常会觉得措手不及。这是一个普遍存在的痛点，无论是在保研面试还是在工作面试中都非常突出。目前市场上缺乏一个专门针对这个问题的工具，用户只能通过学长的帮助或者自己摸索来解决，这个过程既不高效也不系统化。因此，我的产品旨在提供一个系统化的工具，帮助用户针对过往经历和项目进行深入的追问和挖掘，模拟真实的面试场景，并提供具体的改进指导，帮助用户更好地准备面试。


## 发现的痛点和核心灵感
面试中非常核心的一个部分就是对于过往项目和经历的询问，这在计算机领域的面试和保研面试中尤为突出。

在真实的面试中，这一部分常常成为比较大的挑战，尤其是当面试官深入挖掘细节时，很多同学会觉得措手不及。

这个问题无法通过背诵八股和刷题来解决，并且在真的被问住之前很多同学根本不清楚自己欠缺的东西是什么。这无论是在保研还是在工作面试中都是一个普遍的且未被满足的需求，这正是我这个产品要解决的问题。

用户的另一个痛点是不知道自己面试时说的话会让面试官怎么想，有时候一句自以为很不错的表述可能会让面试官觉得很糟糕，用户完全没有上帝视角来判断哪些表述是安全的，哪些表述是高危的，这也是我这个产品要解决的另一个问题。

在我的产品中，给用户实时提供了一个上帝视角，让用户知道自己的每个回答给面试官造成的印象，这样用户在真实的面试中就可以更好地把握面试官的心理和关注点，避免说出一些高危表述。

## 用户调查
我进行下面几个方面的用户调查：
1. 询问了目前在美团任职的学长，了解到他在面试中遇到过被面试官追问简历的情况。
2. 询问了曾经面试过我的面试官（同时也是学长），了解到他自己也在面试中被深入追问简历中项目，并且认为如果准备不善很可能会出问题。
3. 询问了和我同届正在保研的同学，了解到他们在保研面试中也遇到过被追问项目经历细节的情况，并且现在主要靠学长的帮助来解决。
4. 我在小红书上广泛阅读了有关大厂面试相关的帖子，从面试官的角度看到了很多认为“面试者的经历写的很fancy但是根本说不清楚细节”的吐槽，从面试者的角度看到了很多被追问经历细节的分享。
以上的调查让我相信这个问题是一个普遍存在的痛点，是一个真实的用户需求。

以上的调查实际上还非常不充分，后续如果有充分时间我将会进行更大规模的调查来验证这个需求的普遍性和迫切性。

另外在调查过程中值得注意的一点是，你无法通过询问用户需要什么样的产品来真正了解用户的需求，因为用户往往无法想象一个全新的产品是什么样子的，他们只能基于自己已有的经验来描述他们的需求，这就像乔布斯所说的“人们不知道他们需要什么，直到你把它展示给他们看”。

因此，我在做出了可以使用的成品之后，我又找了一些我的朋友进行测试，他们的评价总体是正向的，具体来说，他们认为提问比较彻底、和此前保研时的提问很像、给的建议很有用等等。问及付费意愿时，他们表示如果自己有面试需求，可能会进行付费。

当然，由于短时间内我找到的肯定是我的朋友，因此其评价可能会有一定的偏向性，后续如果有充分时间我将会进行更大规模的测试来验证这个产品的价值和用户的付费意愿。

## 核心受众
本产品第一版的核心受众有以下两类：
1. CS领域的求职者：尤其是那些正在准备技术面试的求职者，他们需要一个工具来帮助他们更好地准备和练习项目相关的问题。
   - 这个群体通常具有较高的付费意愿，同时他们尝试新工具的意愿较强，很适合发展为初期用户。
2. 保研面试的学生：尤其是参加重视科研的项目的学生。
   - 学生群体对于升学相关问题的需求比较迫切，且付费意愿较高。
   - 学生群体的社交网络传播能力较强，非常适合通过口碑传播来扩大用户基础。

后续的用户群拓展：
实际上任何重视过往项目经历的面试者都可以成为潜在用户。这点我已经通过其他行业的简历验证了这一点。但在第一版中，我们将重点优化以上两类客户的体验。因为其他行业对于过往项目的追问可能不一定有这两类群体那么详细。

## 产品的核心创新点
1. 逐项深入追问：不是一个泛泛而谈的面试官，而是针对用户的每个过往经历和项目分别进行深入的追问和挖掘，在其中融入“反事实追问”等追问方式，模拟真实的面试场景。
2. 面试官手稿：实时显示一个面试官手稿，让用户知道自己的每个回答给面试官造成的印象，让用户掌握上帝视角，在真实的面试中可以更好地把握面试官的心理和关注点。
3. 真正的进化指南：在用户完成一个项目的”拷打“后，可以得到一个非常具体的改进指南，告诉用户这个项目是否需要修改简历上的高危表述、需要去额外准备哪几个技术细节，需要去准备哪些相关的追问等等。用户可以得到最直接的改进指导，而不是一个泛泛而谈的得分。
4. 完整的训练循环：用户可以对于一个项目反复训练，每轮都可以看到自己的进步。

## 市场调查
目前已经有很多做AI面试官的产品了，但是其大多数存在下面的问题：
1. 过度关注考题，很多以押题为招牌。实际上在计算机领域，考题早有成熟的解决方案（例如leecode），这并不是用户的真实痛点。
2. 对于过往经历的挖掘比较浅显。很多产品只是简单地根据简历进行一些肤浅的追问，而没有真正深入挖掘用户的过往经历和项目细节。并且也会缺失真实面试场景中的“贡献质疑“、”背景追问“等重要的追问方式。
3. 缺乏真正的改进指导。很多产品只是给出一个模糊的得分或者一些泛泛而谈的建议，而没有提供一个具体的、可操作的、落实到每一条的改进指南，告诉用户具体应该如何改进他们的项目描述和准备。
4. 许多工具在追求大而全，但是实际上一个好的提示词就够把通用的AI模型变成一个非常专业的面试官了。太多没有价值可替代的功能只会稀释用户对于不可替代功能的注意力。

目前这个赛道的玩家主要有：
1. BOSS直聘
2. 智联招聘
3. 牛客
4. OfferGoose 鹅来面
等等

商业调查部分限于时间原因我没有能完全深入的调查，可能会出现遗漏，但总体来说目前这个赛道的玩家大多存在上述问题，并且目前我没有看到一个真正专注于过往经历追问和改进指导的工具。

## 产品设计说明
这个产品的核心功能就是一个过往经历项目的拷打器，用户可以输入自己的简历，系统会自己分析出用户的过往经历和项目，针对每个项目制定追问策略和锚点，在一套面试的方法论下持续对用户进行追问和挖掘（在这个过程中展示面试官的手稿），最终给出一个非常具体的改进指南。用户可以在改进后再次测试，不断迭代，直到自己觉得满意为止。

具体来讲，在v01版本中要实现下面这几个模块：
1. 简历解析模块：从用户的简历中解析出值得追问的项目条目，并且总结整份简历，为后续的追问提供背景。
2. 拷打模块：针对每个项目条目，制定追问策略和锚点，在一套面试的方法论下持续对用户进行追问。
3. 面试官手稿模块：在追问的过程中实时展示一个面试官手稿，让用户知道自己的每个回答给面试官造成的印象。（这个模块与上个模块在代码中是放在一起设计的）
4. 改进指南模块：在用户完成一个项目的”拷打“后，给出一个非常具体的改进指南，告诉用户这个项目是否需要修改简历上的高危表述、需要去额外准备哪几个技术细节，需要去准备哪些相关的追问等等。
5. 训练循环模块：用户可以对于一个项目反复重新拷打，每轮都可以看到自己的进步。


## 目前的已知问题（时间问题尚未修复，在大部分时候不影响使用）
1. 对于少部分PDF格式的简历会存在OCR错误，这个是上游API提供商的问题，我下一步会写一个兜底机制，当解析失败时启用本地的OCR解析。（本地解析在大部分时候不如上游API提供商的解析准确，因此不是默认）
2. 总结模块有一定概率会出现json解析失败的问题，这个问题下一步我会将json的解析工具做的更健壮一些，并且增加重试机制来降低这个问题的发生率。但是第一版为了减少复杂度和时间考虑，我没有进行修复。
3. 如果在模型正在输出时离开这个页面，去其他页面，会造成这段对话历史丢失，且再次进入这个页面时会话历史无法恢复，这个问题下一步我会增加一个本地的缓存机制来解决这个问题。由于发现的较晚，这个修复没有最终部署到服务器上。
4. 如果面试官一次性问了两个相关问题，用户大篇幅的讨论第二个问题而忽略第一个，有较小的概率会让面试官忽略第一个问题，尽管第一个问题可能很重要。这是一个比较隐蔽的bug，下一步我会增加一个机制来确保面试官在问了多个问题时能够兼顾每个问题。由于这个问题比较隐蔽，并且在大部分时候不会出现，因此我发现较晚，因此这一版没有修复。
5. 如果多人同时使用，限于目前的任务队列机制，可能会存在较大的延迟问题。下一步我会优化任务队列机制，改善多用户同时使用时的延迟问题。由于这个问题在单人使用时不会出现，因此我发现较晚，因此这一版没有修复。
