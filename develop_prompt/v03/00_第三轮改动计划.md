# 项目拷打器 · v03 第三轮改动计划:极简登录

> v02 已完成。本轮只加**一个极简登录**:账号名 + 密码,登录后体验与现在**完全一致**,每个用户只能看到自己的简历与项目。**不做注册**,两个账号密码写死。最小改动。

## 核心思路(为什么改动极小)
现有系统已用请求头 `X-Session-Id` 做**按会话的数据隔离**(`resumes.session_id` 及其级联)。v03 只需把「随机匿名会话」换成「登录后拿到的、与用户绑定的不透明令牌」:
- 后端写死两个用户,每个用户一个**不透明的 session 令牌**;登录校验账号密码后把该令牌返回。
- 前端登录成功后用该令牌作为 `X-Session-Id` 发送 → **现有按 session 的隔离逻辑原样生效,即变成按用户隔离**。
- 后端把 `require_session` 从「有头即可」收紧为「必须是已知用户的令牌,否则 401」,从而强制登录、且天然隔离。

结论:**数据模型、BE1/2/3、拷问/指南逻辑全部不改**;后端只加「鉴权 + 登录端点」,前端只加「登录页 + 鉴权守卫」。

## 写死的两个账号(请按此实现,凭此登录)
| 用户名 | 密码 | 内部不透明令牌(session_id,仅后端用) |
|---|---|---|
| `admin` | `kaoda-2025-admin` | `sess-admin-9f3c1a2b8e7d` |
| `guest` | `kaoda-2025-guest` | `sess-guest-4a6d2e9c1b07` |

> 凭据写死在后端代码常量(`backend/app/auth.py`)里——这是最「最小改动」的「写死」方式,无需建用户表/迁移,同样满足「两个固定账号、无注册」。令牌是登录成功后才下发的随机串,起到 bearer 令牌作用,不可猜测。

## 契约增量(向后兼容,唯一跨端新增)
1. 新增 `POST /api/login`,body `{ "username": string, "password": string }`:
   - 成功 200:`{ "session_id": string }`(该用户的不透明令牌)。
   - 失败 401:`{ "error": { "code": "invalid_credentials", "message": "账号或密码错误" } }`。
2. `require_session` 行为收紧:缺失或非已知用户令牌 → 401 `{ "error": { "code": "unauthorized", "message": "未登录或登录已失效" } }`(原为 400 `missing_session`)。其余所有受保护端点不变。

## 改动 → 归属(仅 2 个 agent)
| 区域 | 改动 | 归属 |
|---|---|---|
| 后端鉴权 | `auth.py` 写死用户与令牌;`POST /api/login`;`require_session` 收紧为校验已知令牌 | **BE0 v03** |
| 前端登录 | `lib/session` 改为存登录令牌(不再自动生成);`/login` 路由 + 鉴权守卫;登录页;顶栏登出;类型/endpoints/mock | **FE0 v03** |
| 前端落地 | 登录后据**该用户现存简历**联动:有简历→信息按钮可点且落地信息页;无简历→保持现状(信息按钮禁用、自动落 `/upload`) | **FE0 v03** |

## 登录后落地与信息页按钮(按用户现存简历)——本轮要点
v02 现有逻辑已正确:`App.tsx` 的 `Header` 据 `listResumes()` 算 `hasResumes`,`infoDisabled = hasResumes === false`(空简历禁用信息按钮);`InfoPage` 的 `/` 无参跳转:有简历→`/resume/:id`,确认空→`/upload`。**v03 只需保证这套逻辑跑在「已登录用户」的数据上、并在登录/换号后正确重算**:
- 期望行为:登录后,**库里已有简历的用户**——信息页按钮**直接可点**、落地在信息页,不必先传一份;**没有简历的用户**——保持现状:信息按钮不可点、进入时自动在上传页。
- 关键正确性:登录成功必须**先存令牌再跳转**,使随后的 `listResumes()` 带上该用户令牌;`Header` 仅在已登录时渲染(每次登录全新挂载,`hasResumes` 按当前用户重拉,绝不跨用户残留);`listResumes()` 失败(含 401)**不判为「无简历」**,避免误禁用。

两者只通过上面的契约耦合,可并行;前端 `VITE_USE_MOCK=true` 下用 mock 登录独立开发。

## 执行顺序
- BE0 v03 与 FE0 v03 可并行。前端先行也行(mock 登录用同一组写死账号)。

## 已做的设计决策
- **凭据存代码常量**而非 DB(最小改动;如确需入库可后续平滑扩展)。
- 登录令牌 = 该用户固定的不透明串;前端存 `localStorage`,作为 `X-Session-Id`。刷新保持登录;登出清除。
- **不迁移**旧匿名数据:登录后是干净的按用户空间;两个新账号初始为空。
- 加一个极简登出(便于切换两个账号验证隔离)。

## 本轮不改
- 不动数据模型与所有业务端点逻辑、不动 SSE、不动 BE1/2/3、不动 v01/v02 既有功能。
- 不做注册、找回密码、多设备、权限分级、用户表/迁移。
