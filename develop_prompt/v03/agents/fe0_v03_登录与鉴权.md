## 改动清单

### 1) `lib/session.ts`(改为登录令牌模型,不再自动生成)
- `getSessionId()`:返回 `localStorage["session_id"]` 或空串(**删除自动生成逻辑**)。
- 新增 `isLoggedIn(): boolean`(令牌非空)、`setSession(token: string)`、`clearSession()`。
- 保留 v02 的 `getLastResumeId/setLastResumeId`;登出时一并清除 last_resume_id。

### 2) `api/types.ts`
```ts
export interface LoginRequest { username: string; password: string; }
export interface LoginResponse { session_id: string; }
```

### 3) `api/endpoints.ts`
```ts
export function login(username: string, password: string): Promise<LoginResponse> {
  if (USE_MOCK) return mock.login(username, password);
  return apiFetch('/api/login', { method: 'POST', body: JSON.stringify({ username, password }) });
}
```
> `apiFetch` 会注入 `X-Session-Id`;登录时该头为空无妨(后端 `/login` 公开)。

### 4) Mock(`mocks/handlers.ts`)——用同一组写死账号校验
```ts
const MOCK_USERS: Record<string, { password: string; session_id: string }> = {
  admin: { password: 'kaoda-2025-admin', session_id: 'sess-admin-9f3c1a2b8e7d' },
  guest: { password: 'kaoda-2025-guest', session_id: 'sess-guest-4a6d2e9c1b07' },
};
export function login(username: string, password: string): Promise<LoginResponse> {
  const u = MOCK_USERS[username];
  if (!u || u.password !== password) return reject('invalid_credentials', '账号或密码错误');
  return sleep(120).then(() => ({ session_id: u.session_id }));
}
```
> mock 其余 handler 不做按用户隔离(全局态),真实后端才隔离——这是已知且可接受的简化,FE 验收以登录/守卫流程为准。

### 5) 登录页 `pages/LoginPage.tsx`(新建)
- 全屏居中、留白充裕的极简卡片:品牌徽标 + 两个输入框(用户名/密码,图标前缀、`aria-label`、密码用 `type=password`)+ 一个主提交按钮(复用 `Button`)。
- 提交:调 `login()` → 成功 `setSession(res.session_id)` 并 `navigate('/')`;失败以克制视觉表达「账号或密码错误」(轻微抖动/红描边即可,尽量少文案)。
- 风格与现有设计系统(暖白纸感、暖色点缀、衬线标题、柔影、温柔动效)一致;回车提交。

### 6) `App.tsx`:路由守卫 + 公开登录路由 + 顶栏登出
- 新增公开路由 `/login` → `LoginPage`,**且登录页用独立的全屏布局(不渲染主应用顶栏的功能图标)**。
- 其余所有路由用守卫:未 `isLoggedIn()` → `<Navigate to="/login" replace />`。可用一个 `RequireAuth` 包裹受保护区域;`Header`(品牌 + v02 的信息/上传图标)仅在已登录时渲染。
- 顶栏右侧加一个克制的**登出**图标按钮:`clearSession()` 后 `navigate('/login')`(便于切换 admin/guest 验证隔离)。
- (可选,体验更稳)在 `client.ts` 的 `apiFetch`/`apiSSE` 遇到 401 时 `clearSession()` 并跳 `/login`;若实现,保持最小。

### 7) 登录后落地与信息页按钮:按「该用户现存简历」联动(必须正确)
`App.tsx` 的 `Header` 已据 `listResumes()` 算 `hasResumes` 并以 `infoDisabled = hasResumes === false` 禁用信息按钮;`InfoPage` 的 `/` 无参 effect 已做「有简历→`/resume/:id`,确认空→`/upload`」。**这套逻辑保留,你只需保证它跑在已登录用户的数据上、并在登录/换号后正确重算**:
- **目标行为**:登录后,**库里已有简历的用户**——信息页按钮**直接可点**、落地在信息页(经 `/`→`/resume/:id`),不必先上传一份;**没有简历的用户**——保持现状:信息按钮不可点、进入时自动落到 `/upload`。
- **登录顺序**:登录成功必须**先 `setSession(token)` 再 `navigate('/')`**,使随后的 `listResumes()` 带上该用户令牌;不要在令牌写入前发起受保护请求。
- **按用户重算、不残留**:`Header` 仅在已登录时渲染(第 6 点),从而每次登录都是全新挂载、`hasResumes` 从该用户重新拉取;登出→换号登录后,信息按钮与落地都基于新用户的现存简历重算,绝不跨用户残留(如把 `hasResumes` 之类状态做成随登录态/`Header` 挂载而重置)。
- **失败不误禁用**:`listResumes()` 失败(含 401)保持现有「不判为无简历(`hasResumes` 维持 `null`,不禁用)」的处理,避免把有简历的用户误禁。
- 不需要改 `InfoPage` 的 `/` 跳转与边栏逻辑本身——它已正确;本条是确保 v03 的会话/守卫改动不破坏它,并在登录后按当前用户重算。
