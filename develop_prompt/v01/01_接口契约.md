# 项目拷打器 · v01 接口契约(唯一事实来源)

> 本文件是前后端之间**唯一**的耦合点,在编码前已冻结。后端按此实现,前端按此 mock 与联调。
> 任何变更必须先改本文件并同步两端。涉及实体见 §6 数据模型,接口见 §2~§4,流式见 §5。

## 0. 全局约定

- 所有接口前缀 `/api`。
- 所有请求必须带请求头 `X-Session-Id: <uuid>`(前端首访生成并存 localStorage)。后端据此隔离数据。缺失时返回 400 `missing_session`。
- 所有 ID 为字符串(uuid4)。时间为 ISO8601 UTC 字符串(如 `2026-05-24T08:00:00Z`)。
- 统一错误响应体:
  ```json
  { "error": { "code": "string", "message": "string" } }
  ```
  HTTP 状态:400 入参错误 / 404 不存在 / 409 状态冲突 / 500 内部错误(含 LLM/Doc2X 失败)。
- 所有 JSON 字段命名用 `snake_case`。

## 1. 状态机(枚举)

```
ResumeStatus        = parsing | parsed | failed
InterrogationStatus = preprocessing | ready | in_progress | ended | failed
GuideStatus         = generating | ready | failed
TrafficLight        = red | yellow | green
ManuscriptKind      = first_impression | reaction | closing
TodoCategory        = resume_fix | knowledge_prep | other
```

状态流转:
- Resume:`parsing` →(双 LLM 完成)`parsed`;失败 `failed`。`parsed` 时自动为每个项目创建 round=1 的 interrogation 并触发预处理。
- Interrogation:`preprocessing` →(初印象+首问生成完)`ready` →(用户答第一题起)`in_progress` →(模型判定结束)`ended`;任一步失败 `failed`。
- Guide:interrogation 进入 `ended` 时自动创建并 `generating` →(生成完)`ready`;失败 `failed`。

## 2. M1 简历解析

### POST `/api/resumes` — 提交简历(文本或 PDF)
两种 Content-Type 二选一:

**文本方式** `application/json`:
```json
{ "text": "粘贴的简历全文..." }
```
**PDF 方式** `multipart/form-data`:字段 `file`(单个 PDF)。

**响应 202**:
```json
{ "resume_id": "uuid" }
```
后端行为:若 PDF 先经 Doc2X 转文本;立即返回,后台并行调用两次 LLM(摘要 + 项目),完成后置 `parsed` 并为每个项目建 round=1 interrogation + 触发预处理。

错误:400 `empty_input`(文本空 / 无文件)、400 `bad_file`(非 PDF)、500 `pdf_failed`。

### GET `/api/resumes/{resume_id}` — 轮询解析结果
**响应 200** `ResumeDetail`:
```json
{
  "id": "uuid",
  "status": "parsing | parsed | failed",
  "summary": {
    "headline": "应届硕士 · 计算机视觉方向",
    "items": [
      { "label": "姓名", "value": "张三" },
      { "label": "学历", "value": "某大学 硕士在读" },
      { "label": "专业", "value": "计算机科学与技术" },
      { "label": "方向", "value": "图像分割 / 模型部署" }
    ]
  },
  "projects": [
    {
      "id": "uuid",
      "name": "基于 X 的实时分割系统",
      "raw_description": "简历原文中该项目的完整描述...",
      "order_index": 0,
      "current_interrogation": { "id": "uuid", "status": "preprocessing | ready | ..." }
    }
  ],
  "error": null
}
```
- `status=parsing` 时 `summary=null`、`projects=[]`;前端轮询(建议 1.2s 间隔)直到 `parsed`/`failed`。
- `current_interrogation` 为该项目最新一轮的 interrogation 概要(用于前端直接跳转拷问页)。

## 3. M2 拷问 + 面试官手稿

### GET `/api/projects/{project_id}` — 项目详情与历史
**响应 200**:
```json
{
  "id": "uuid",
  "name": "基于 X 的实时分割系统",
  "raw_description": "...",
  "order_index": 0,
  "current_interrogation_id": "uuid",
  "interrogations": [
    { "id": "uuid", "round_number": 2, "status": "ended", "ended": true,
      "has_guide": true, "created_at": "..." },
    { "id": "uuid", "round_number": 1, "status": "ended", "ended": true,
      "has_guide": true, "created_at": "..." }
  ]
}
```
`interrogations` 按 `round_number` 倒序(最新在前)。

### GET `/api/interrogations/{interrogation_id}` — 拷问详情(渲染拷问页)
**响应 200** `InterrogationDetail`:
```json
{
  "id": "uuid",
  "project_id": "uuid",
  "round_number": 1,
  "status": "preprocessing | ready | in_progress | ended | failed",
  "ended": false,
  "prev_round_number": null,
  "model": "claude-opus-4-6",
  "manuscript": [
    { "index": 0, "kind": "first_impression", "content": "**初印象**\n该项目描述用词较大...", "created_at": "..." }
  ],
  "turns": [
    { "index": 0, "question": "你提到实时,具体端到端延迟是多少?在什么硬件上测的?", "answer": null, "created_at": "..." }
  ],
  "closing_message": null,
  "error": null
}
```
- `status=preprocessing` 时 `manuscript=[]`、`turns=[]`;前端轮询直到 `ready`。
- `ready` 时:`manuscript[0]` 为初印象,`turns[0].question` 为首问且 `answer=null`(待答)。
- 历史轮:`turns` 中每条都有 `answer`;`ended=true` 时 `closing_message` 为礼貌结束语。

### POST `/api/interrogations/{interrogation_id}/answer` — 提交回答(**SSE 流式**)
**请求** `application/json`:
```json
{ "turn_index": 0, "answer": "端到端延迟约 80ms,在 RTX 3060 上用 batch=1 实测..." }
```
- `turn_index` 必须等于当前最后一个未回答的 turn 的 index,否则 409 `turn_conflict`。
- 仅当 interrogation 处于 `ready` 或 `in_progress` 时可调用,否则 409 `not_answerable`。
**响应**:`Content-Type: text/event-stream`,见 §5。后端先落库 answer(并把 status 置 `in_progress`),再流式生成「手稿增量 + 下一问 / 结束语」,流结束时落库;若结束则置 `ended` 并自动触发指南生成。

### POST `/api/projects/{project_id}/reinterrogate` — 再次拷打(新一轮)
**请求** `application/json`(均可选):
```json
{ "model": "gpt-5.5" }
```
**响应 202**:
```json
{ "interrogation_id": "uuid" }
```
后端:`round_number = 当前最大 + 1`;`prev_summary = 最近一轮 ready 指南的 summary_for_next`(无则 null);`status=preprocessing` 并触发预处理。旧轮次保留。前端随后轮询新 interrogation 至 `ready`。

### GET `/api/models` — 可选模型列表(供模型切换器)
**响应 200**:
```json
{ "models": ["claude-opus-4-6", "gpt-5.5", "gemini-2.5-pro"], "default": "claude-opus-4-6" }
```

## 4. M3 总结 / 改进指南

### GET `/api/interrogations/{interrogation_id}/guide` — 改进指南(轮询)
**响应 200** `GuideDetail`:
```json
{
  "id": "uuid",
  "interrogation_id": "uuid",
  "status": "generating | ready | failed",
  "traffic_light": "yellow",
  "overview": "整体框架清晰,但‘实时’与‘高并发’两处表述需要更扎实的数据支撑。",
  "todos": [
    { "id": "uuid", "category": "resume_fix",
      "content": "把‘实现高并发实时分割’改为给出具体 QPS 与延迟数字的客观表述", "done": false, "order_index": 0 },
    { "id": "uuid", "category": "knowledge_prep",
      "content": "准备:为何选 TensorRT 而非 ONNXRuntime,二者在该场景的量化精度/吞吐差异", "done": false, "order_index": 1 },
    { "id": "uuid", "category": "other",
      "content": "复盘当被追问‘你个人具体贡献’时的回答,准备可量化的个人产出", "done": false, "order_index": 2 }
  ],
  "error": null
}
```
- `status=generating` 时 `traffic_light/overview=null`、`todos=[]`;前端轮询直到 `ready`/`failed`。
- 响应**不含** `summary_for_next`(内部字段,仅用于下一轮拷问)。
- 404 `guide_not_found`:interrogation 尚未结束(未触发生成)。

### PATCH `/api/todos/{todo_id}` — 勾选/取消 todo
**请求**:`{ "done": true }` → **响应 200**:更新后的 todo 对象(同上结构)。

## 5. SSE 流式协议(仅 POST `.../answer`)

每个事件为标准 SSE:`event: <名>\n` + `data: <JSON 单行>\n\n`。

| 事件 | data | 说明 |
|---|---|---|
| `manuscript_delta` | `{"text":"片段"}` | 面试官手稿(本轮新增条目)的增量,多次 |
| `question_delta` | `{"text":"片段"}` | 下一个问题的增量,多次(仅当未结束) |
| `closing_delta` | `{"text":"片段"}` | 结束礼貌语的增量,多次(仅当结束) |
| `done` | `{"ended":bool,"manuscript_index":int,"next_turn_index":int\|null}` | 收尾;落库完成。`ended=true` 时 `next_turn_index=null` |
| `error` | `{"code":"...","message":"..."}` | 出错,流终止 |

**典型未结束轮**:`manuscript_delta`×N → `question_delta`×N → `done{ended:false, manuscript_index:k, next_turn_index:k}`
**结束轮**:`manuscript_delta`×N → `closing_delta`×N → `done{ended:true, manuscript_index:k, next_turn_index:null}`

> 索引规则:用户回答 turn[i] 后,生成的手稿条目 index = i+1,下一问 turn index = i+1。前端可凭 `done` 的索引与本地状态对齐,或在流结束后重新 GET interrogation 做一致性兜底。

### 5.1 模型流式输出的内部分段格式(后端解析依据)
拷问提示词要求模型**严格**按以下顺序、用独占一行的分段标记输出(`结束` 标记在 `问题` 之前,以便后端在流出问题前就知道是否结束):
```
[手稿]
<面试官手稿增量,中文 markdown>
[结束]
true 或 false
[问题]
<下一个要问的问题;若结束,则为礼貌结束语>
```
后端状态机:读到 `[手稿]` 后的文本 → `manuscript_delta`;读到 `[结束]` 解析布尔;读到 `[问题]` 后的文本 → 若未结束发 `question_delta`,若结束发 `closing_delta`。预处理(初印象+首问)用同一格式但非流式,后台解析后落库。

## 6. 数据模型(SQLAlchemy / SQLite)

> 仅列字段与关系;类型用通用记法。所有表含 `id`(str, PK)与 `created_at`(datetime)。

### resumes
| 字段 | 类型 | 说明 |
|---|---|---|
| session_id | str, index | 匿名会话 |
| source | str | `text` \| `pdf` |
| raw_text | text | 粘贴或 Doc2X 转出的全文 |
| status | str | ResumeStatus |
| summary_json | JSON, null | `{headline, items:[{label,value}]}` |
| error | str, null | |

### projects(归属 resume)
| 字段 | 类型 | 说明 |
|---|---|---|
| resume_id | FK → resumes | |
| name | str | 项目/经历名称 |
| raw_description | text | 简历原文描述(不二次总结) |
| order_index | int | 展示顺序 |

### interrogations(归属 project,一次「拷打」)
| 字段 | 类型 | 说明 |
|---|---|---|
| project_id | FK → projects | |
| round_number | int | 从 1 起 |
| status | str | InterrogationStatus |
| prev_summary | text, null | 上一轮的 `summary_for_next`,首轮 null |
| model | str | 使用的模型名 |
| ended | bool | |
| closing_message | text, null | 结束礼貌语 |
| error | str, null | |

### manuscript_entries(归属 interrogation,面试官手稿条目)
| 字段 | 类型 | 说明 |
|---|---|---|
| interrogation_id | FK | |
| index | int | 0 起;0=初印象 |
| kind | str | ManuscriptKind |
| content | text | 中文 markdown |

### turns(归属 interrogation,一问一答)
| 字段 | 类型 | 说明 |
|---|---|---|
| interrogation_id | FK | |
| index | int | 0 起 |
| question | text | 面试官问题 |
| answer | text, null | 用户回答,未答为 null |

### guides(归属 interrogation,1:1)
| 字段 | 类型 | 说明 |
|---|---|---|
| interrogation_id | FK, unique | |
| status | str | GuideStatus |
| traffic_light | str, null | TrafficLight |
| overview | text, null | 面向用户的一句话总览 |
| summary_for_next | text, null | **内部**:给下一轮拷问的拷打摘要 |
| error | str, null | |

### todos(归属 guide)
| 字段 | 类型 | 说明 |
|---|---|---|
| guide_id | FK → guides | |
| category | str | TodoCategory |
| content | text | 可执行的一条 todo |
| done | bool | 默认 false |
| order_index | int | |

## 7. 关系总览
```
resume 1──* project 1──* interrogation 1──* turn
                                  │        1──* manuscript_entry
                                  └─ 1──1 guide 1──* todo
```

## 8. 后端 LLM 调用一览(实现参考)
| 模块 | 提示词文件 | 模式 | 输入 | 输出 |
|---|---|---|---|---|
| M1 摘要 | `resume_summary.md` | JSON 非流式 | 简历全文 | `{headline, items[]}` |
| M1 项目 | `resume_projects.md` | JSON 非流式 | 简历全文 | `{projects:[{name, raw_description}]}` |
| M2 预处理 | `interrogation.md` | 分段格式 非流式 | 摘要+项目+prev_summary | 初印象手稿 + 首问 |
| M2 对话轮 | `interrogation.md` | 分段格式 **流式** | 上同 + 历史问答 + 本次回答 | 手稿增量 + 下一问/结束 |
| M3 指南 | `guide.md` | JSON 非流式 | 摘要+项目+全手稿+全问答+prev_summary | `{traffic_light, overview, todos[], summary_for_next}` |

M1 的两次调用**并行**(asyncio.gather)。预处理对所有项目**并行**触发。
