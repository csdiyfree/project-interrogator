# 项目拷打器 · v02 第二轮改动计划

> v01 已完成并在本机部署测试。本轮针对 `ideas/第二轮改进计划/改进计划.md` 的 6 条反馈做**最小改动**。
> 原则:① 改动最小、不重构;② 契约只做**向后兼容的增量**;③ 全部落到现有 8 个 agent(BE0-3 / FE0-3),**无需新增 agent**(BE3 后端本轮不动)。
> 每条改动对应 `agents/` 下一份**增量提示词**,你把它整段发给对应角色的新 agent 即可;提示词会指引它先读现有代码再改。

## 已锁定的产品决策(2026-05-24)
- **路由**:`/` = 信息页;`/upload` = 独立上传页,上传解析完成后**自动跳转** `/resume/:resumeId`;`/resume/:resumeId` = 指定简历信息页。前端用 localStorage 记住上次打开的简历。
- **简历命名**:自动名 = `{候选人姓名} {上传时间}`;同会话内重名再追加 ` (1)`、` (2)`;无姓名时回退 headline,再回退日期。均可手动改名。
- **PDF 解析卡顿**:只改前端——提交后立即进入全屏「处理中」动画,覆盖较慢的上传/Doc2X POST 与解析轮询;**后端不动**。

## 契约增量(唯一跨端耦合点,全部向后兼容)
1. `ResumeDetail` 增加字段 `name: string`。
2. 新增 `GET /api/resumes` → `{ "resumes": ResumeListItem[] }`,`ResumeListItem = { id, name, status, created_at }`,按 `created_at` 倒序,按 `X-Session-Id` 过滤。
3. 新增 `PATCH /api/resumes/{resume_id}`,body `{ "name": string }`(非空),返回更新后的 `ResumeListItem`。
4. `ProjectDetail` 增加字段 `resume_id: string`。

> 其余 5 项改进都在既有契约内完成(导航、动画、粘附滚动、项目切换、再次拷打"装作不知道")。本计划即增量契约的事实来源;BE0 落地到 `schemas.py`、FE0 落地到 `api/types.ts`。

## 执行顺序(两条轨道并行;每条内部地基先行)
- 后端:**BE0**(模型+schema 增量)先做 → 然后 **BE1**、**BE2** 并行。
- 前端:**FE0**(路由/顶栏/类型/endpoints/mock/占位页)先做 → 然后 **FE1**、**FE2**、**FE3** 并行。
- 前端靠 FE0 更新后的 mock 层独立联调,不依赖后端进度。

## 本地数据库说明(重要)
BE0 给 `resumes` 表新增 `name` 列。应用用 `Base.metadata.create_all`,**不会**给已存在的表加列。本机现有 `backend/data/app.db` 需二选一:
- 保数据:执行 `ALTER TABLE resumes ADD COLUMN name VARCHAR;`
- 弃数据(推荐,演示库可丢):删除 `backend/data/app.db`,重启后端自动重建。

## 本轮不改
- 不动 BE3(指南后端)、不动 SSE 协议、不动评分/总结逻辑、不引入鉴权/迁移框架/队列。
- 不做简历删除、不做跨轮进度看板、不做语音(仍属后续版本)。
